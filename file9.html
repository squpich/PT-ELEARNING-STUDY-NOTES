1. O que se estuda neste ficheiro

O TP10 continua o bloco tecnológico da disciplina e explica em detalhe os standards de integração de conteúdos de e-learning: SCORM, xAPI, cmi5 e LTI. O ficheiro também aborda os princípios de interatividade, estratégias ativas, Web 2.0, communication tools, Web 3.0 e mobile apps.

Ideia principal: um módulo de e-learning de qualidade não deve apenas parecer um site, mas também ser compatível com plataformas, rastreável, interativo, pedagogicamente significativo e adequado à integração num ecossistema educativo digital.

SCORM

Standard clássico de empacotamento, lançamento e tracking de conteúdos de e-learning em LMS.

xAPI

Standard flexível para registar learning experiences em diferentes contextos, não apenas dentro de uma LMS.

cmi5

Perfil xAPI para LMS, que define regras de lançamento, autorização e monitorização.

LTI

Standard para ligar ferramentas educativas externas a LMS através de integração segura.

Interactivity

A interatividade deve apoiar objetivos de aprendizagem, autonomy, feedback e active participation.

Web tools

Web 2.0, communication tools, Web 3.0 e mobile apps expandem o ambiente de e-learning.

2. Desenvolvimento histórico das plataformas digitais

A apresentação mostra o desenvolvimento das plataformas de e-learning através de três etapas: domínio da web, surgimento de standards/open-source e fase atual de recursos, integração, cloud, mobile devices, analytics, personalization e AI.

Web

Os cursos passaram a estar disponíveis através do browser, substituindo gradualmente o CD-ROM. Isto ampliou o público e levou à primeira geração de LMS comerciais baseadas na web, que geriam users, content, assessment e communication.

2000–2002

Surge a primeira versão do SCORM em 2000. Em 2002 surge o Moodle como primeira LMS open-source, reforçando o papel dos standards e do open-source no e-learning.

Etapa atual

Cloud e mobile devices alteram as necessidades dos utilizadores. Surgem advanced analytics, xAPI, personalization e AI como novas áreas de desenvolvimento da aprendizagem digital.

Lógica de desenvolvimento: primeiro era importante simplesmente lançar o curso no browser. Depois passou a ser importante transferir o curso entre plataformas. Hoje é importante acompanhar a learning experience em diferentes ambientes e personalizar a aprendizagem.

3. SCORM — Shareable Content Object Reference Model

SCORM é um conjunto de especificações técnicas criado para garantir interoperability entre conteúdos de e-learning e diferentes plataformas. Permite que materiais educativos sejam reutilizados e lançados em várias LMS sem adaptações específicas.

Definição curta: SCORM é um standard de empacotamento, lançamento, comunicação e tracking de conteúdos de e-learning dentro de uma LMS.

Porque o SCORM é importante

  • Antes do SCORM, o conteúdo criado para uma plataforma frequentemente funcionava mal noutra.
  • A ausência de um standard comum aumentava a dependência de fornecedores específicos de software.
  • Recriar o mesmo curso para diferentes sistemas aumentava os development costs.
  • SCORM criou uma forma comum de packaging, communication e tracking.
  • As organizações passaram a recolher dados mais estáveis sobre progress, completion e performance.
  • SCORM tornou-se o standard dominante de distribuição de digital learning content durante mais de uma década.

Como funciona o SCORM

ZIP package

O curso é distribuído como pacote ZIP com todos os ficheiros e metadata.

imsmanifest.xml

Índice principal do curso: structure, resources e navigation rules.

JavaScript API

O conteúdo comunica com a LMS através de uma JavaScript API standardizada.

Tracking

Normalmente são registados completion status, score, time e learner location.

Authoring tool

O conteúdo é criado numa authoring tool e depois exportado em SCORM.

LMS execution

Uma LMS compatível lança o pacote e recebe dados sobre a realização.

Evolução e legado do SCORM

  • SCORM 1.2 tornou-se a versão mais amplamente adotada devido a simplicity, stability e compatibility.
  • O principal legado do SCORM é um interoperable ecosystem para distribuição de educational content.
  • A standardização do tracking permitiu acompanhar completion, progress e performance de forma comparável.
  • As limitações do SCORM em mobile learning, offline learning e data model levaram ao desenvolvimento de xAPI e cmi5.

4. xAPI — Experience API

xAPI, inicialmente conhecido como Project Tin Can, foi criado para recolher e armazenar dados sobre learning experiences em diferentes contextos. Foi publicado em 2013 como resposta às novas necessidades do digital learning.

Definição curta: xAPI regista experiências educativas não apenas em LMS, mas também em mobile apps, simulations, games, ambientes presenciais e outros contextos.

Porque o xAPI se tornou necessário

  • O crescimento de mobile apps, serious games, simulations e VR exigiu tracking mais flexível.
  • Tornou-se necessário acompanhar experiências de aprendizagem em diferentes devices e contexts, não apenas no browser e na LMS.
  • As organizações passaram a querer richer indicators: engagement, performance, collaboration, informal learning.
  • xAPI pode registar online e offline activities.
  • xAPI ajuda a construir lifelong learning records.
  • Supera LMS-centered tracking limitations dos modelos tradicionais.

Como funciona o xAPI

Actor

Quem realizou a ação: estudante, colaborador, utilizador.

Verb

O que fez: completed, answered, watched, attempted, passed, shared.

Object

Sobre o que foi a ação: lesson, video, quiz, simulation, activity.

Context

Informação adicional: duration, date, result, source validation, environment.

LRS

Learning Record Store recolhe, armazena e disponibiliza learning records.

Analytics

Os dados são usados para reports, personalization, learning analytics e predictive analysis.

Vantagens do xAPI

  • Flexibilidade: permite acompanhar quase qualquer learning experience.
  • Suporte a online e offline learning com sincronização posterior.
  • Integração de LMS, mobile apps, enterprise systems e analytics tools.
  • Desenvolvimento de Learning Record Stores, reporting tools e specialized profiles.
  • Suporte a personalization e data-based learning.
Exemplo de xAPI statement: “Student completed Quiz 1” ou “Student watched Introduction Video”. Qualquer evento xAPI é construído segundo o esquema: actor + verb + object.

5. cmi5 — perfil xAPI para LMS

cmi5 é um perfil xAPI criado especificamente para ambientes LMS. Define regras standardizadas para launch, authentication e monitoring of educational content.

Definição curta: cmi5 liga a flexibilidade do xAPI à lógica tradicional de lançamento e tracking de cursos em LMS.

Para que serve o cmi5

  • xAPI é muito flexível, mas sem regras adicionais pode ser demasiado livre para LMS.
  • cmi5 define regras consistentes para lançamento e monitoring de conteúdo em LMS.
  • Não substitui xAPI; especifica a sua aplicação a LMS-distributed courses.
  • O seu desenvolvimento está ligado às limitações do SCORM em mobility, authentication e restrictive data models.

Como funciona o cmi5

cmi5.xml

Descreve course structure, available modules e launch rules.

Assignable Units

AUs — unidades de aprendizagem que comunicam com a LMS através de xAPI.

LMS launch

A LMS transmite identity, LRS link e secure authentication parameters.

Lifecycle statements

As AUs enviam dados: started, completed, passed, failed, terminated.

LRS records

Os registos são armazenados no LRS e usados para progress e results.

LMS decision

A LMS lê dados do LRS para determinar course completion e learner status.

Vantagens do cmi5

  • Estrutura mais rigorosa do que xAPI puro.
  • Mais flexível do que SCORM, porque é baseado em xAPI.
  • Adequado para distributed content, simulations e adaptive modules.
  • Considera melhor authentication, launch e LMS reporting.
  • É estrategicamente importante como ponte entre o modelo clássico de LMS e o moderno learning analytics.

6. LTI — Learning Tools Interoperability

LTI é um standard 1EdTech para integração de applications e services externos com learning platforms, por exemplo LMS. Ao contrário de SCORM e xAPI, LTI não se foca em conteúdo ou records of learning, mas na ligação entre sistemas independentes.

Definição curta: LTI permite lançar ferramentas educativas externas diretamente a partir da LMS através de integração segura e Single Sign-On.

Como funciona o LTI

  • LTI liga Tool Consumer e Tool Provider.
  • Tool Consumer — plataforma recetora, por exemplo LMS.
  • Tool Provider — ferramenta externa: simulator, videoconference system, assessment platform, virtual lab.
  • O utilizador clica numa ligação na LMS para aceder à ferramenta externa.
  • A LMS envia secure request com credentials e contextual data.
  • Podem ser transmitidos name, email, user role, course context e grade information.
  • As versões novas suportam Deep Linking, grade synchronization e automatic participant management.

Porque o LTI é importante

  • Complementa SCORM, xAPI e cmi5.
  • SCORM distribui content, xAPI records experiences, enquanto LTI liga systems and tools.
  • Permite construir modular educational ecosystems.
  • Reduz development costs e diminui a dependência de proprietary integrations.
  • Aumenta technological flexibility e apoia innovation.
  • É um dos principais mecanismos de interoperability entre LMS e specialized services.

7. Comparação entre SCORM, xAPI, cmi5 e LTI

Standard Foco principal Onde funciona O que regista / faz Principal limitação
SCORM Empacotamento e lançamento de conteúdo. Dentro de LMS. Completion, score, time, location. Limitado a LMS, browser, cenários online e data model rígido.
xAPI Registo de learning experiences. LMS, apps, games, simulations, offline/online contexts. Actor-Verb-Object statements + context/result. Exige LRS e uma arquitetura de dados mais complexa.
cmi5 xAPI em contexto LMS. LMS + LRS. Launch, authentication, AU lifecycle, completion. Menos difundido e exige infraestrutura compatível.
LTI Integração de external tools. LMS + external applications. Secure launch, SSO, contextual data, grade sync. Não é standard de empacotamento de conteúdo nem learning record completo.
Em resumo: SCORM — curso como pacote. xAPI — experiência como dados. cmi5 — xAPI para LMS. LTI — ferramentas externas dentro de LMS.

8. Princípios de interatividade

A interatividade no e-learning deve ser pedagogicamente significativa. A ação do utilizador deve ajudar understanding, application ou consolidation of knowledge.

Alignment

Cada ação do utilizador deve estar ligada aos learning objectives.

Autonomy

O estudante recebe control, choice e oportunidades de exploration.

Decision-making

Uma boa interatividade obriga a escolher, experimentar, comparar e aplicar.

Immediate feedback

O estudante deve compreender rapidamente as consequências das suas ações e corrigir erros.

Meaningful feedback

O feedback deve explicar porque a resposta está correta ou incorreta.

Appropriate challenge

A dificuldade deve corresponder ao nível de competência dos estudantes.

Ambiente interativo eficaz

Os digital interactive environments mais eficazes combinam active participation, meaningful feedback, intuitive navigation e challenges adequados ao nível dos estudantes.

Pergunta de verificação: esta ação ajuda o estudante a compreender, aplicar ou consolidar conhecimento? Se não, não é interatividade educativa, mas um clique decorativo.

9. Atividade “Operação Engagement”

O ficheiro inclui uma atividade prática de 60 minutos: os estudantes, em grupos de 2–3 pessoas, recebem um documento sobre ergonomia e devem transformá-lo num recurso multimédia mais attractive e interactive para o target audience.

O que deve ser apresentado

  • A concept idea desenvolvida.
  • As principais proposed interactions.
  • A aplicação de principles of interactivity.
  • Um exemplo de active participation.
Importante: não é necessário desenvolver tecnicamente o recurso. O objetivo da atividade é projetar a learning experience, e não o produto final.

10. Plataformas Web 2.0

As plataformas Web 2.0 apoiam collaborative learning, shared production, discussion e active participation. São especialmente úteis quando o estudante deve não apenas consumir material, mas criar, comentar, comparar e colaborar.

Plataforma Para que é adequada Benefício educativo
Wiki Criação e edição colaborativa de uma knowledge base. Collective construction of knowledge, documentation, glossary.
Padlet Quadro visual com notas, links, imagens e ideias. Brainstorming, sharing resources, recolha rápida de ideias.
Miro Collaborative whiteboard para esquemas, maps, workflows e design thinking. Group planning, concept maps, storyboard, process visualization.
Flipgrid Respostas em vídeo e curtas student presentations. Oral expression, reflection, peer feedback, social presence.
Notion Notes, databases, project documentation e shared workspace. Organization, collaborative documentation, project management.

Risco da Web 2.0

Se não forem definidas regras, papéis e objetivo da atividade, a Web 2.0 transforma-se numa troca caótica de links. A plataforma deve apoiar uma learning activity concreta.

11. Ferramentas de Comunicação

Communication tools apoiam comunicação síncrona e assíncrona, group coordination, feedback e acompanhamento em online learning.

Discord

Canais, voice rooms, community interaction, discussões rápidas e apoio informal.

Microsoft Teams

Videoconferências, ficheiros, turmas, assignments, integração com Microsoft 365.

Slack

Canais de equipa, threads, integrations, project communication e professional workflow.

Como utilizar sem caos

  • Definir onde se colocam perguntas sobre conteúdo e onde se discute o projeto.
  • Separar canais por temas.
  • Fixar instruções e prazos.
  • Não duplicar a mesma informação em vários canais.
  • Utilizar notifications conscientemente para não criar overload.

12. Web 3.0 no e-learning

Web 3.0 no contexto da disciplina está sobretudo ligada a intelligent systems: educational chatbots, virtual tutors e predictive analytics. Estas ferramentas permitem reforçar personalization, acompanhamento e data-based decision-making.

Educational chatbots

Respondem a perguntas frequentes, ajudam a orientar-se no módulo e dão apoio básico.

Virtual tutors

Podem acompanhar o estudante, sugerir pistas e adaptar explicações.

Predictive analytics

Usa dados para identificar risco de dropout, pontos fracos e necessidade de intervention.

Riscos da Web 3.0

  • A AI pode dar respostas imprecisas ou inadequadas.
  • São necessários privacy, data protection e transparência no uso dos dados.
  • O estudante não deve substituir totalmente o pensamento crítico por uma sugestão automática.
  • Predictive analytics pode ser biased se os dados forem incompletos ou mal interpretados.

13. Mobile Apps

Mobile apps apoiam learning on the move, microlearning, repetition, gamification e verificações interativas rápidas. Mas o formato móvel exige blocos curtos, clear UI e responsive design.

Aplicação Função na aprendizagem O que se pode retirar para o próprio projeto
Kahoot! Quizzes de jogo e verificação rápida da compreensão. Perguntas curtas, competição, immediate feedback.
Duolingo Microlearning, streaks, gamification, tarefas curtas. Progress, pequenas aulas, repetição, motivação visível.
Codecademy Go Prática de programação e repetição do material no telemóvel. Mini-exercícios, consolidação, learning by doing.
Elevate Treino de cognitive skills através de exercícios curtos. Atividades curtas baseadas em challenge e progresso personalizado.
Para e-learning: mobile não significa “o mesmo site, só mais pequeno”. São necessários blocos curtos, touch targets grandes, navegação simples e ação rápida.

14. Erros frequentes

Erro 1: confundir standards: SCORM não é o mesmo que xAPI, cmi5 ou LTI.

Erro 2: considerar SCORM ultrapassado e inútil. Continua a ser amplamente utilizado.

Erro 3: pensar que xAPI funciona apenas em LMS. Pelo contrário, serve para diferentes contextos.

Erro 4: esquecer que xAPI exige LRS.

Erro 5: interpretar cmi5 como substituto de xAPI. É um perfil xAPI para LMS.

Erro 6: considerar LTI um standard de empacotamento de conteúdo. LTI é integration entre systems/tools.

Erro 7: adicionar interatividade por cliques, e não por learning objectives.

Erro 8: utilizar muitas communication tools sem regras.

Erro 9: transferir conteúdo desktop longo para mobile sem adaptação.

Erro 10: não explicar na defesa porque a tecnologia escolhida é adequada ao módulo.

Perguntas de autoavaliação

1. Para que serve o SCORM?
Para empacotar, lançar, reutilizar e acompanhar conteúdos de e-learning em LMS.

2. O que existe dentro de um pacote SCORM?
ZIP com ficheiros do curso, metadata e imsmanifest.xml, que descreve a estrutura e a navegação.

3. Em que é que xAPI difere de SCORM?
SCORM funciona sobretudo dentro de LMS; xAPI pode registar learning experiences em diferentes ambientes.

4. O que é Actor-Verb-Object?
Estrutura de um xAPI statement: quem fez, o que fez e sobre que objeto.

5. O que é LRS?
Learning Record Store — sistema de armazenamento e disponibilização de learning records.

6. O que é cmi5?
Perfil xAPI para LMS, que define regras de lançamento, autorização e monitoring.

7. O que é uma Assignable Unit?
Unidade de aprendizagem em cmi5 que é lançada a partir da LMS e transmite xAPI statements.

8. Para que serve LTI?
Para ligar external tools a LMS de forma segura, incluindo SSO e transmissão de contexto.

9. O que distingue uma boa interatividade?
Está ligada a objectives, oferece autonomy, decision-making e meaningful immediate feedback.

10. O que é Operação Engagement?
Atividade em que um grupo transforma um documento num multimedia learning resource mais atrativo e interativo.

11. Como Web 3.0 é aplicada no e-learning?
Através de chatbots, virtual tutors e predictive analytics.

12. Conclusão principal do TP10?
As tecnologias de e-learning devem garantir compatibilidade, tracking, integration, interactivity e pedagogical value.

Conclusão final: TP10 mostra a arquitetura técnica do e-learning moderno: SCORM responde pelo pacote e LMS-tracking, xAPI pelo learning record amplo, cmi5 pelo xAPI dentro de LMS, LTI pela ligação de ferramentas externas, enquanto a interatividade e Web tools transformam o conteúdo numa experiência de aprendizagem ativa.
